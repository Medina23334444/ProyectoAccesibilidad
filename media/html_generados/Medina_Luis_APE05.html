FEIRNNR Carrera de Computación ● Plantillas ADR (Architectural Decision Records): Para documentar decisiones de forma estructurada. ● Procedimiento / Metodología Ejecutada Metodología: Aprendizaje Basado en Proyectos (ABPr). El procedimiento de la práctica consistió en la transformación de los requisitos de accesibilidad del entorno universitario (CIM) hacia una arquitectura técnica específica (PSM) mediante el modelado en StarUML, donde se definieron estilos de Microservicios y Cliente-Servidor para cumplir con el rendimiento y la escalabilidad necesarios. Se aplicaron patrones de diseño como Strategy para la conversión de formatos (HTML/EPUB) y Adapter para la integración de validadores externos (WAVE), estructurando el sistema en paquetes de Frontend (React), Backend (Django 3.11) e infraestructura de contenedores Docker. Finalmente, se validó la integridad del diseño mediante una tabla de trazabilidad que asegura que el procesamiento de datos se realice en memoria volátil, garantizando el cumplimiento ético de no persistencia de archivos en el servidor (RNF04 ● Resultados Estilos arquitectónicos aplicados • ADR 01: Cliente-Servidor Contexto: La Universidad Nacional de Loja requiere un prototipo de accesibilidad digital sea accesible desde navegadores web. El sistema debe procesar archivos PDF cargados por el usuario para convertir a formatos como paginas HTML o EPUB. Decisión: Se adopta un estilo Cliente-Servidor. El cliente será una interfaz web ligera y el servidor centralizará la lógica de conversión. Justificación: Se reduce la carga de procesamiento en el dispositivo del usuario al centralizar la lógica de conversión en el servidor. De este modo, la complejidad del procesamiento del PDF, como el análisis de objetos, la extracción de metadatos y la validación de las reglas WCAG, no depende del rendimiento del hardware local del usuario. Garantizando que se cumpla el requisito no funcional de rendimiento, procesar archivos de 5MB en menos de 3 minutos. Consecuencias: • Se requiere una conexión estable a internet. • Requiere protocolos de red para una comulación segura (TLS/SSL), para proteger él envió de documentos académicos. • ADR 02: Microservicios Contexto: El proceso de conversión semántica es intensivo en recursos y debe cumplir con el RNF05 de rendimiento (máximo 3 minutos para 5MB). Además, debe integrarse con validadores externos como WAVE y EPUBCheck.

FEIRNNR Carrera de Computación Decisión: Descomponer el servidor en Microservicios especializados: uno para la "Conversión PDF" y otro para la "Validación de Accesibilidad". Justificación: Al separar la lógica de negocio, se garantiza que una caída de las herramientas externas de validación no bloquee el núcleo de conversión del sistema. Además, permite este estilo permite integrar nuevos formatos de salida o herramientas de validación sin alterar la arquitectura base. Consecuencias: • Incremento en la complejidad de red debido a la comunicación entre servicios mediante APIs, requiriendo mayor monitoreo. Patrones arquitectónicos aplicados • ADR 03: Strategy Contexto: El prototipo debe permitir la conversión de un mismo origen un archivo PDF a múltiples formatos de salida (paginas HTML o EPUB). Cada formato requiere algoritmos y reglas de accesibilidad distintas. Decisión: Aplicar el Patrón Strategy. Se define una interfaz común para la conversión, y cada formato (paginas HTML y EPUB) implementa su propia estrategia algorítmica. Justificación: Permite intercambiar el algoritmo de conversión en tiempo de ejecución según la elección del usuario (RF02/RF03). Facilita la extensibilidad, permitiendo añadir nuevos formatos (como Audio) sin modificar la arquitectura base. Consecuencias: • Requiere dedicar un tiempo extra a la fase inicial de análisis para asegurar que la interfaz Strategy sea lo suficientemente genérica para todos los formatos. • ADR 04: Adapter Contexto: El prototipo depende de APIs externas de terceros para validar la accesibilidad. Estas APIs tienen sus propios formatos de entrada y salida que no coinciden con el modelo interno del prototipo. Decisión: Utilizar el patrón Adapter. Se define una interfaz interna estandarizada para la Validación de Accesibilidad, y se crea un adaptador específico para cada herramienta externa que traduce las peticiones de nuestro prototipo al formato requerido por el tercero. Justificación: Actúa como un traductor entre el núcleo de nuestra aplicación y las herramientas externas (WAVE, EPUBCheck). Si el proveedor externo cambia su API, solo se modifica el Adapter, protegiendo la lógica de negocio principal. Consecuencias: • Añade una pequeña sobrecarga de código inicial, pero simplifica drásticamente el mantenimiento futuro.

Tecnología: La arquitectura de Microservicios desplegada en contenedores Docker. Tiene una ventaja en relación al principio ético de la honestidad. Al finalizar la tarea, el contenedor puede ser reiniciado, garantizando que los archivos no se carguen permanentemente en el servidor y se eliminen automáticamente tras su uso. Esta es una ventaja ética superior frente a sistemas que suelen dejar rastros de datos en carpetas temporales del disco duro. ● Conclusiones Se logró una alineación del 70 % entre los requisitos funcionales (RF01RF06) y el diseño técnico, utilizando los patrones Strategy y Adapter para asegurar que la lógica de conversión semántica sea modular.
